2026-06-03

🔴 Critical: 3 · 🟡 Important: 4 · 🟢 Opportunities: 5

Primary: Jakub Krehel — Production Polish

The Neo experience has strong creative ambition — the 300vh scrollytelling hero, the 3D unbox module, and the Vibe guide character are genuinely distinctive. But below those showcase moments the site behaves like a static document: seven full-width content blocks stack with no transition, no entrance timing, and no visual grouping. Theme switching snaps. Subpage navigation snaps. Interactive buttons give no tactile response. The polish investment is heavily front-loaded in the first two sections and then disappears. The biggest win isn't adding more animation — it's adding entrance continuity to what already exists, and restructuring the content blocks so users understand what groupings mean before they scroll through them.

Issues to Address

🔴 Critical

GSAP and Three.js ignore prefers-reduced-motion

The CSS @media (prefers-reduced-motion: reduce) rule correctly disables CSS keyframes, but the Three.js WebGL tick loops in NeoHero and TrustHero run unconditionally — 60fps particle scatter and rotation continues regardless of user preference. The NeoGuide GSAP character also runs its mousemove + scroll animations with no reduced-motion check. For users with vestibular disorders, a spinning 3D particle cloud is a trigger, not a detail.

app.jsx:863 (NeoHero tick loop) · app.jsx:1074 (TrustHero tick loop) · app.jsx:204 (NeoGuide)

Fix: Check window.matchMedia('(prefers-reduced-motion: reduce)').matches before starting both Three.js tick loops. For reduced-motion: render a single static frame and stop the RAF. The unbox already does this correctly at app.jsx:1796 — same pattern needed in heroes.

🔴 Critical

Subpage content swaps with no transition

When a user taps a subnav item, the ternary {active ? <SubpageDetail> : <SectionOverview>} hard-snaps between two substantially different content layouts. There's no opacity fade, no translateY, no blur — the content teleports. This is the most common interaction on any section page and it currently has less polish than a 2010 jQuery tab.

app.jsx:2375

Fix: Wrap in CSS transitions using a key-change pattern: add a unique key to SubpageDetail and SectionOverview, then use CSS animation on mount via a class toggled on first paint. Enter: opacity 0→1 + translateY 10px→0 + blur 3px→0 over 280ms. Exit: opacity 1→0 + translateY 0→-5px over 160ms (subtler, per Jakub).

🔴 Critical

Theme toggle (Neo ↔ Trust) causes full page snap

The theme pill in the nav switches between two completely different design systems — dark neon vs. light enterprise. Currently this is a hard React re-render with no transition. On a site where the entire visual identity changes in one click, the snap is jarring enough to break trust in the product quality. The color, typography, and layout all change simultaneously with zero visual continuity.

app.jsx:1547-1560 (Site render) · app.jsx:1410 (theme pill)

Fix: Add a CSS class .theme-transitioning to .site on toggle, then use a 200ms opacity crossfade. Alternatively: on toggle, snap a screenshot of the current theme, overlay it, then remove the overlay with a 200ms fade to reveal the new theme beneath. Simplest CSS-only approach: transition: background 200ms ease, color 200ms ease on .site and its key color tokens.

🟡 Important

All below-fold sections enter with zero animation

After the 300vh hero and 360vh unbox, the home page has five consecutive full-width sections: SectionCards, SuperSkills, MarketContext, HomeProof, HomeIntakeCTA. All five snap in as the user scrolls to them — no opacity, no translateY, no timing. Given the visual investment in the hero sections, this sudden shift to static-document behavior is a polish cliff. The Vibe character's elaborate scroll reactions also stop at this point, so the energy drop is compounded.

app.jsx:1480-1486

Fix: Add animation-timeline: view() (CSS scroll-driven) to each section with a simple opacity 0→1 + translateY 24px→0 enter. Use animation-range: entry 0% entry 30% so sections reveal in the first third of their scroll entry. No JS needed — pure CSS, respects reduced-motion, and gives native 60fps performance.

Issues to Address

🟡 Important

Buttons have no press feedback — zero :active scale

Every button on the site — hero CTAs, section cards, subnav, contact form submit — has no :active state animation. On a Gen Z site where tactile response is a baseline expectation (every mobile interaction has scale feedback), buttonless `:active` states feel like a prototype, not a shipped product. The Neo CTAs in particular — "Enter the Chaos", "Open the brief" — are brand moments that should feel punchy on press.

base.css:36 (.hero-actions button — no :active)

Fix: Add to base.css: button:active { transform: scale(0.96); transition: transform 80ms cubic-bezier(0.16,1,0.3,1); } — one rule, every button on the site. The release should spring back faster than the press: pair with transition: transform 180ms cubic-bezier(0.16,1,0.3,1) on the base button rule.

🟡 Important

Vibe character has 7+ infinite CSS pulses with no reduced-motion guard in keyframes

The guide character runs neoLedPulse, neoRingPulse, neoCoreBreath, neoIntroPulse, neoEyeBlink, neoLedBlink, and neoHeroFloat — all animation: ... infinite. The global prefers-reduced-motion CSS rule collapses these, which is correct. However, the character is rendered inside the 300vh hero — when the hero is off-screen, all these CSS animations keep running silently on compositor threads. There's no display: none or animation-play-state: paused applied when the character is off-screen.

neo-guide.css:77,104-106,121-122,129,241-242

Fix: The character already has a GSAP autoAlpha: 0 when off-screen (visibility hidden). Add animation-play-state: paused to the guide overlay when autoAlpha is 0 — or use the existing IntersectionObserver on the hero to toggle .is-paused class: .neo-guide-overlay.is-paused * { animation-play-state: paused; }

Issues to Address

🟡 Important

Content block density: 7 sequential sections with no visual grouping

The home page renders: Hero → Unbox → SectionCards → SuperSkills → MarketContext → HomeProof → HomeIntakeCTA. That's 5 full-width content blocks after the hero modules, each separated only by padding. There's no visual hierarchy, no grouping signal, no typographic distinction between "what Nexara does" (SectionCards + SuperSkills) and "why it works" (MarketContext + HomeProof). Users can't orient themselves in the page. Emil's rule: if a user has to scroll through content to understand the structure, the structure is wrong.

app.jsx:1480-1486

Fix — two groups:
Group A: "What we build" — SectionCards + SuperSkills stacked with a shared section heading and a dividing line before the next group.
Group B: "Where and why" — MarketContext + HomeProof side-by-side on desktop (two-col grid) so they read as paired evidence.
HomeIntakeCTA stays as the final single-col CTA block with visual separation (border-top + more vertical space).
This reduces perceived complexity from 5 blocks to 3 groups without removing any content.
